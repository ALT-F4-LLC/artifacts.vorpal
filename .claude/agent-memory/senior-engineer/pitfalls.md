# senior-engineer recurring pitfalls

Append-only. Format: `symptom → root cause → resolution`. Newest at bottom.

- npm-package artifact's language servers crash at runtime with MODULE_NOT_FOUND despite the tarball building fine → published npm tarballs ship `lib/` that `require`s runtime deps but do NOT bundle `node_modules`, and vorpal build steps run OFFLINE (Rust artifact uses `cargo --offline`; no build-step `npm install` exists anywhere in src/artifact/) → vendor the full production dependency closure as individual `ArtifactSource` tarballs from registry.npmjs.org (fetched pre-build, offline-safe) inside the one artifact file, then `cp` them into a flat `node_modules` at the package root during the build step. A flat layout works when the closure is conflict-free (no two deps need incompatible majors). Validate with a build-step LSP `initialize` self-test (pattern: typescript_language_server.rs) — a shallow `--version` check exits before module resolution and misses the failure.
- Prebuilt-binary artifact (Bun/Node/Deno self-contained exe) crashes at startup with ENOENT reading a bundled asset (theme/locale/template JSON) even though the build succeeded → the build step copied ONLY the binary into `bin/`, but these runtimes resolve bundled assets relative to `dirname(process.execPath)`, so the binary looks for `bin/<asset>/...` which doesn't exist. The upstream release tarball ships the binary ALONGSIDE its asset dirs (theme/, assets/, etc.). → Copy the WHOLE upstream tree to `$VORPAL_OUTPUT/lib/<name>` and put a thin `#!/bin/sh` wrapper in `bin/<name>` that `exec`s the real binary by absolute path (so execPath → lib/<name>/<name>, assets resolve at lib/<name>/...). Mirrors typescript_language_server.rs lib+wrapper convention. Don't use a symlink (execPath under a symlink depends on whether the runtime realpaths it — uncertain). Add a build-step smoke test that drives the REAL asset-loading path (`--version` often short-circuits before it; e.g. for a TUI run `<bin> </dev/null` to hit the main/init path) and fails the build on the ENOENT. Verify the actual error path FIRST by extracting the tarball + reading the decompiled resolver (`strings <bin> | grep getThemesDir/getPackageDir`) — OBSERVE, don't infer which assets are needed; preserve the full tree.
- Local Bash-tool shell test wrote a wrapper whose shebang came out as `#\!/bin/sh` (broken; exec → ENOEXEC) → zsh history-expansion mangles `!` even inside single-quoted heredocs fed to `bash -s` in the Bash tool; it can still "pass" via bash's ENOEXEC fallback to /bin/sh, masking the corruption → write test scripts containing `#!/bin/sh` (or any `!`) to a file via the Write tool and run with `bash <file>`, never via an inline zsh heredoc. Note this is a TEST-HARNESS artifact only — the real vorpal artifact is a Rust string run in the VM's non-interactive bash, which does NOT history-expand, so `#!/bin/sh` in pi.rs is fine.
